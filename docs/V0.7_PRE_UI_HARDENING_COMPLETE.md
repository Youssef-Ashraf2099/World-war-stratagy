### 1️⃣ **WARFARE SUBSYSTEM** (`src/subsystems/warfare.rs`)

**Role:** War declarations, peace treaties, conflict resolution  

**Notifications Integrated:**
- `WarDeclared` → Emitted when `declare_war()` is called
  - Recipients: Aggressor nation `+` Defender nation
  - Payload: War ID, casus belli, war goal
  - Timing: Instant (same tick as declaration)

- `WarEnded` → Emitted when `sign_peace()` is called
  - Recipients: Both warring parties
  - Payload: Victor, defeated party, peace terms
  - Timing: Instant (same tick as treaty signature)

**Critical Code Path:**
```rust
pub fn declare_war(world: &mut World, aggressor: Entity, defender: Entity, 
                   casus_belli: String, war_goal: String, tick: u32) {
    // ... validation & war entity creation ...
    create_war_notification(world, aggressor, defender, tick); // ← NEW
}
```

### 2️⃣ **NUCLEAR SUBSYSTEM** (`src/subsystems/nuclear.rs`)

**Role:** Nuclear capability development, treaty compliance, apocalyptic deterrence  

**Notifications Integrated:**
- `NuclearWeaponUsed` → Emitted when `apply_nuclear_use_effects()` triggers
  - Recipients: **ALL NATIONS** (global event)
  - Payload: Attacking nation, target nation, warhead count
  - Timing: Instant
  - **Critical:** Nuclear use is a WORLD awareness event

- `NuclearTreatyViolation` → Emitted when treaty violation detected
  - Recipients: **ALL NATIONS** (enforcement mechanism)
  - Payload: Violating nation, treaty ID
  - Timing: Instant

- `NuclearCapabilityAchieved` → Emitted when nation reaches 30% nuclear readiness
  - Recipients: **ALL NATIONS** (deterrence world knowledge)
  - Payload: Achieving nation
  - Timing: One-time per nation

**Critical Design Decision:**
Nuclear events are **explicitly GLOBAL** because nuclear deterrence is a world-knowledge phenomenon. All nations immediately become aware of capability or weapon use. This is different from warfare events (which notify belligerents) or espionage (which notifies perpetrator+target).

**Test Coverage:**
- `test_regression_nuclear_use_notifies_bystanders` — Validates all-nation notification
- `test_warfare_and_nuclear_actions_emit_notifications` — Integration with warfare

### 5️⃣ **ECONOMIC SUBSYSTEM** (`src/subsystems/economic.rs`)

**Role:** Production, trade, GDP growth, resource management  

**NEW: Crisis Detection Function**
Added `pub fn check_crises(world: &mut World, tick: u32)` to proactively emit crisis notifications:

**Notifications Emitted:**
- `EconomicCrisis` → When nation's GDP drops below 5.0
  - Recipients: Affected nation
  - Threshold: GDP < 5.0
  - Payload: Nation entity, current GDP
  - Timing: Every tick condition is true

- `LegitimacyCrisis` → When nation's legitimacy drops below 25.0
  - Recipients: Affected nation
  - Threshold: Legitimacy < 25.0
  - Payload: Nation entity, current legitimacy
  - Timing: Every tick condition is true

**Integration into Pipeline:**
`check_crises()` is called in the EconomicPhase after GDP updates. Both crisis types can be emitted simultaneously if nation is in severe dual-collapse.

**Test Coverage:**
- `test_economic_phase_updates_gdp_and_stress` — GDP growth validation
- `test_economic_phase_emits_crisis_notifications_for_critical_values` — Crisis emission
- `test_economic_crisis_path_emits_both_crisis_notifications` — Dual-crisis scenario

### 6️⃣ **ALLIANCE SUBSYSTEM** (`src/subsystems/alliance.rs`)

**Role:** Coalition formation, cohesion decay, diplomatic obligations  

**Notifications Integrated:**

- `AllianceFormed` → Emitted when `spawn_alliance()` creates new alliance
  - Recipients: All founding members
  - Payload: Alliance ID, member list, duration
  - Timing: Instant at creation

- `AllianceBroken` → Emitted when `check_dissolutions()` detects dissolution
  - Recipients: All former members (before despawn)
  - Payload: Alliance ID, reason (low cohesion / insufficient members)
  - Timing: Last tick before entity despawn

**Critical Implementation Detail:**
Alliance dissolution now:
1. Notifies all members **before** despawn
2. Despawns the Alliance entity
3. Cleans up BiMap associations

This ensures members can react to alliance failure (e.g., seeking new allies) before entity is gone.

**Test Coverage:**
- `test_alliance_dissolution_notifies_members_and_despawns_alliance` — Full dissolution flow

## 🏗️ ARCHITECTURE READINESS

### Notification System Flow

```
[Subsystem Event]
    ↓
[Check Condition] (e.g., war declared, nation in crisis)
    ↓
[create_*_notification(world, ...)] → Creates Notification entity
    ↓
[Notification stored to NotificationLog component]
    ↓
[UI can query NotificationLog to display to player]
    ↓
[Notifications pruned when log > 1000 entries]
```

### V0.7 Pipeline (19 Phases)

```
1. AdvancedAIDecision      7. ReligionConflict         13. Technology
2. DiplomaticContact       8. Diplomacy                14. Intelligence  
3. Warfare (📢 emit WarDeclared/WarEnded)         15. Culture
4. Nuclear (📢 emit NuclearWeaponUsed globally)  16. Environment
5. Vassalage (📢 emit Vassalization/Rebellion)    17. Society
6. Espionage (📢 emit EspionageRevealed)         18. Prosperity
7. Alliance (📢 emit AllianceBroken on dissolve)  19. NotificationPhase (📢 Aggregates)
   Economic (📢 emit EconomicCrisis/Legitimacy)
```

**Critical:** Phase ordering ensures notifications emit when subsystem completes, before aggregation in final phase.

## 🎓 LESSONS LEARNED

1. **Reactive Systems Require Regression Guards**
   - Notification emission is easy to accidentally break
   - Specific guards on event-type emission are critical
   - Two new regression tests added to prevent future removal

2. **Long-Run Tests Expose Subtle Issues**
   - Resource NaN, population underflow, and log explosion only visible at 1000+ ticks
   - Stress tests at 30 nations revealed log pruning effectiveness
   - Performance baseline validated (<15ms/tick maintained)

3. **Notification Scoping Matters**
   - GLOBAL (nuclear) vs BILATERAL (espionage) vs MULTILATERAL (alliance) have different semantics
   - Incorrect scoping breaks gameplay mechanics (e.g., deterrence)
   - Explicit test coverage for each scoping type is essential

4. **Crisis Detection Must Be Proactive**
   - Economic crises aren't just penalties; they're player-visible events
   - Both EconomicCrisis and LegitimacyCrisis can emit simultaneously
   - Proper threshold logic prevents false positives/negatives
